## 1. 产品概述

### 1.1 项目背景
随着地理信息应用（GIS）在智慧城市、交通监控、环境监测、军事仿真等领域的深入发展，各类业务系统对底图数据及三维模型数据的统一管理与分发需求日益迫切。当前需要构建一套独立、高可用、标准化的地图瓦片服务系统，统一管理二维栅格瓦片（影像、矢量底图）与三维瓦片（3D Tiles），并提供标准化的访问接口与可视化监控能力。

### 1.2 产品定义
本系统是一个**私有化/云端部署的地图瓦片分发与监控平台**。它负责接收并存储通过离线工具或自动化流程生成的XYZ栅格瓦片数据与3D Tiles三维数据；通过统一的RESTful API对外提供服务；采用API Key机制进行访问鉴权；提供后台管理界面用于数据源配置、Key管理和调用统计可视化。

### 1.3 核心价值
- **统一分发**：兼容二维地图（WebGIS）与三维场景（Cesium/UE）的瓦片服务，一套系统支撑多端应用。
- **安全可控**：通过Key鉴权防止数据盗用，支持Referer白名单，保障地理信息资产安全。
- **可观测性**：实时统计各业务方（Key）的调用量，为计费、限流或容量规划提供数据支持。

### 1.4 目标用户

| 用户角色 | 描述 | 核心诉求 |
| :--- | :--- | :--- |
| **系统管理员** | 负责系统的部署、运维与数据管理 | 数据上传/更新、创建API Key、查看全系统调用趋势、监控服务健康状态。 |
| **业务开发者** | 调用瓦片服务的第三方系统开发人员 | 获取有效的API Key，查阅接口文档，快速接入二维/三维瓦片服务。 |
| **运营/产品人员** | 关注数据使用情况与成本 | 查看每日调用统计报表，分析各业务线使用量，进行数据付费结算依据。 |

## 2. 详细功能列表

### 2.1 服务端功能

#### 2.1.1 数据源管理
- **数据源注册**：支持注册不同类型的瓦片数据集，包括：
  - **XYZ栅格瓦片集**：需指定存储路径（本地磁盘、S3、OSS）、瓦片格式（PNG/JPEG/WebP）、最小/最大层级（z范围）。
  - **3D Tiles数据源**：需指定入口文件（`tileset.json`）的存储路径。
- **数据源列表**：展示所有已发布的数据集，支持按类型（2D/3D）筛选，支持启用/禁用（禁用后无法访问）。
- **数据更新**：支持通过文件覆盖方式增量更新瓦片（需维护瓦片服务缓存一致性）。
-  **瓦片预览**：支持地图瓦片的预览功能，二维地图可用leaflet,三维地图用Cesium实现。

#### 2.1.2 授权管理
- **API Key生成**：系统自动生成32位字符串作为唯一Key，支持自定义备注（如“某某项目生产环境”）。
- **权限绑定**：
  - 为每个Key勾选允许访问的数据源（可多选）。
  - 设置IP白名单（可选，支持CIDR格式）。
  - 设置Referer白名单（可选，防盗链）。
- **Key状态管理**：支持启用、禁用、删除操作。删除后立即失效。

#### 2.1.3 瓦片服务接口
- **标准XYZ服务**：符合 `/{dataset_id}/{z}/{x}/{y}.{format}` 或 `/{key}/{dataset_id}/{z}/{x}/{y}.{format}` 的RESTful风格。
- **3D Tiles服务**：提供 `/{dataset_id}/tileset.json` 入口，支持动态解析内部子资源的路径重定向（需处理相对路径与Key鉴权）。
- **服务发现**：提供根目录接口，列出当前Key有权限访问的所有数据集列表及其类型、基本信息。

#### 2.1.4 统计监控
- **日志采集**：记录每一次瓦片请求，包括：时间戳、API Key、数据集ID、瓦片坐标、响应状态码、响应时间、IP地址、User-Agent。
- **实时统计**：
  - 今日总调用量、昨日总调用量、本月累计调用量。
  - 按Key维度展示调用量排名。
  - 按数据集维度展示调用量排名。
- **历史报表**：
  - 支持按日、按周、按月查询调用趋势图（折线图/柱状图）。
  - 支持导出Excel/CSV报表。
  - 支持失败率统计（4xx、5xx错误占比）。

### 2.2 后台管理界面（Web端）

- **登录认证**：基于RBAC的角色控制，仅管理员可登录。
- **仪表盘**：展示关键指标（请求总数、活跃Key数、服务状态）。
- **数据源管理页面**：表单式添加/编辑数据源，支持预览数据源（如展示一个简单的XYZ瓦片预览图或加载3D Tiles简单查看器）。
- **授权管理页面**：管理API Key，清晰展示Key的创建时间、最后调用时间、调用次数。
- **统计报表页面**：ECharts图表展示调用趋势，支持日期范围选择器，支持表格导出。

## 4. 界面设计要求

### 4.1 设计原则
- **简洁高效**：后台管理界面采用侧边栏导航，主内容区采用卡片式布局，减少视觉干扰。
- **数据可视化**：统计数据尽量用图表展示，减少纯数字罗列。
- **响应式**：虽然主要为PC后台使用，但需适配1920*1080及1366*768分辨率。

### 4.2 关键页面原型说明

1.  **登录页**：
    - 品牌Logo居中，用户名+密码输入框。
    - 需包含“记住我”选项。

2.  **仪表盘**：
    - 顶部：今日总请求数、昨日总请求数、在线Key数、服务状态（绿/红）。
    - 中间：近7天请求趋势（折线图）。
    - 底部：Top 5 调用Key排行榜（表格）。

3.  **数据源管理**：
    - 列表展示：名称、类型（XYZ/3DTiles）、瓦片格式、状态（启用/禁用）、操作。
    - **新增/编辑弹窗**：表单包含名称、类型（下拉）、存储方式（本地路径/S3路径）、瓦片格式（自动识别或手动选择）。
    - **预览操作**：对于XYZ，弹窗展示简单的OpenLayers地图加载该源；对于3DTiles，嵌入Cesium Viewer加载该源。

4.  **授权管理**：
    - 列表展示：Key（脱敏显示前6后4）、备注、绑定数据源、状态、最后调用时间、操作（禁用/删除）。
    - **新增Key弹窗**：填写备注，多选框选择该Key可访问的数据源，填写IP白名单（支持多行输入）。

5.  **统计报表**：
    - 顶部日期范围选择器（支持快捷选今日/本周/本月）。
    - 图表区：调用趋势图（可按“总调用”或“按Key”分组查看）。
    - 数据明细表：支持按Key、按数据集维度下钻，支持导出Excel。

## 5. 技术栈建议

为实现高并发瓦片分发、灵活部署，建议采用以下技术栈：

| 层级          | 技术选型                    | 说明                                                                               |     |
| :---------- | :---------------------- | :------------------------------------------------------------------------------- | --- |
| **后端语言**    | **Go**                  | Go语言在高并发IO场景下内存占用更低，适合纯静态资源代理；Spring Boot生态成熟，适合复杂业务逻辑。建议选 **Go + Gin** 以降低资源成本。 |     |
| **存储（元数据）** | **PostgreSQL**          | 用于存储数据源配置、API Key信息以及海量调用日志。                                                     |     |
| **缓存**      | **Redis**               | 用于缓存API Key权限信息（减少DB查询），以及缓存热点瓦片（尤其是`tileset.json`这类频繁访问但体积小的文件）。                |     |
| **瓦片存储**    | **文件系统**                | 初期支持本地文件系统。                                                                      |     |
| **管理后台前端**  | **Vue3 + Element Plus** | 成熟的组件库，快速搭建后台管理界面。                                                               |     |
| **数据可视化**   | **ECharts**             | 实现调用趋势图、饼图等统计展示。                                                                 |     |
| **网关/负载**   | **Nginx**               | 置于服务前端，处理Gzip压缩、HTTPS卸载、静态资源缓存。对于纯静态瓦片，可使用Nginx的`try_files`直接返回，减少对应用服务的压力。      |     |

## 6. 非功能性需求

### 6.1 性能需求
- **并发能力**：单节点需支持 **1000+ QPS** 的瓦片请求（假设每个瓦片大小50KB~200KB）。
- **响应时间**：瓦片接口平均响应时间 < 100ms (排除网络延迟)。
- **吞吐量**：系统需支持至少 **500 MB/s** 的出口带宽利用。
- **冷启动**：服务重启后，鉴权缓存预热应在10秒内完成。

### 6.2 安全需求
- **鉴权隔离**：所有瓦片请求必须携带有效Key（Query参数：`?key=xxx` 或 Header: `Authorization: Bearer xxx`），未授权请求返回403。
- **越权防护**：Key仅能访问被授权的数据集，后端需在中间件层校验数据集ID是否在权限列表中。
- **防攻击**：
  - 针对高频恶意请求，需支持基于Key或IP的限流（Rate Limit），例如每秒限制100次。
  - 接口防SQL注入（参数化查询）。
- **传输安全**：生产环境强制启用HTTPS，TLS版本不低于1.2。

### 6.3 可用性与可靠性
- **高可用**：服务端支持无状态水平扩展，多节点部署时使用Redis共享缓存与权限数据。
- **数据一致性**：当数据源存储（如瓦片文件）更新时，需支持刷新Nginx缓存（如通过脚本清除特定URL的缓存），避免返回旧数据。
- **故障恢复**：服务崩溃时，具备自动重启能力（如使用Supervisor或K8s的Deployment）。

### 6.4 可扩展性
- **存储扩展**：设计`StorageAdapter`接口，后续新增云存储（如阿里云OSS、AWS S3）时无需修改业务逻辑。
- **格式扩展**：接口设计预留`format`参数，未来支持矢量瓦片（MVT）时，仅需增加新的路由处理器。

### 6.5 可维护性
- **日志**：输出结构化日志（JSON格式），便于接入ELK（Elasticsearch, Logstash, Kibana）或Loki日志系统。
- **监控**：提供 `/metrics` 端点，暴露服务运行指标（QPS、错误数、延迟），供Prometheus采集，并在Grafana展示。
